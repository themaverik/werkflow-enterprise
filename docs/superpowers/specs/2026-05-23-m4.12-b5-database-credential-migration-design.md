# M4.12 Phase B.5 — Database Credential Migration (Design)

**Date:** 2026-05-23
**Milestone:** M4.12 — Connector Architecture Maturity, Phase B.5
**Scope:** Backend only. Portal credential picker split to a follow-up B.5-portal slice.
**ADR:** Amends `docs/adr/ADR-020-credential-types-as-peer-concept.md`.

## Goal

Migrate tenant **database** datasource passwords off the admin-side shadow AES-256-GCM
store (`tenant_datasource.encrypted_password` via `EncryptionService`) onto the
OpenBao-backed `CredentialType` pattern already used by HTTP credentials (B.2–B.4),
implementing the forward-looking `DatabaseCredentialType` interface shipped in B.4.

This removes the last plaintext-credential-over-internal-HTTP path: today the engine's
`DatasourceRegistry` asks admin's `resolveForEngine`, which decrypts the password and
returns it as plaintext. After B.5 the engine resolves the credential itself from OpenBao,
exactly as the HTTP delegates do.

## Current State (verified)

- `tenant_datasource` (admin, created V20, renamed V22): columns include plaintext
  `username` and AES ciphertext `encrypted_password`. **No SQL seed inserts into this
  table** — rows are runtime-created only; in practice 0–1 throwaway rows (a hand-registered
  demo H2 HRIS using `sa`/empty). The V21 demo connector references `datasourceRef:
  "demo-h2-hris"` but never seeds the datasource row itself.
- `EncryptionService` (`com.werkflow.common.security`, single master key
  `WERKFLOW_MASTER_ENCRYPTION_KEY`) is also used by `ConnectorService` for
  `TenantApiCredential.secretValue` (connector API keys). **It must stay** — B.5 only
  removes its use by `TenantDatasourceService`. (Connector API-credential migration is a
  separate future phase, out of B.5 scope.)
- Engine `DatasourceRegistry.createPool` builds the HikariCP pool, reading
  `username`/`password` from `DatasourceConfigDto` (fetched from admin, password decrypted).
- HTTP credential pattern: `tenant_credentials` is metadata-only
  (`(tenant_id, credential_type, label) -> vault_path`); admin `TenantCredentialService` is
  a dumb metadata+vault store writing opaque `{values}` maps to OpenBao via
  `VaultCredentialStore`; engine `CredentialRegistry.resolveForTenant(type, tenantId, label)`
  asks admin for the path (`CredentialMetadataClient`, 60s `@Cacheable` TTL) and reads OpenBao
  via `VaultReader`. CredentialType impls (name/displayName/fields/validate) live only in
  engine at `services/engine/.../action/credential/types/`.

## Locked Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| D1 | Username lives **in** the credential (username+password pair), not on the datasource | Matches `CredentialValues` design; a DB login is a pair that rotates together; credential becomes self-contained and reusable across datasources |
| D2 | **TRUNCATE**, no automated backfill | 0–1 throwaway H2 rows (`sa`/empty); a backfill job would be fragile run-once throwaway code. Existing data is disposable (confirmed) |
| D3 | Engine resolves **OpenBao-direct** via `CredentialRegistry`; admin's `resolveForEngine` drops the plaintext password | Removes last plaintext-over-HTTP path; mirrors HTTP delegates |
| D4 | **Portal out of scope** — B.5-portal follow-up slice | Mirrors B.4's proven rhythm; backend reviewable/mergeable alone |
| D5 | `testConnection` **reads vault in admin** (not delegated to engine) | Most surgical; reuses existing SSRF + error-categorization logic; admin already has vault read access from B.2 |
| D6 | Interface becomes `applyCredentials(HikariConfig, CredentialValues)`, not `buildDataSource(...) -> DataSource` | Keeps `poolName`/`autoCommit`/cache/circuit-breaker policy in `DatasourceRegistry` (the only place with tenant context); matches the HTTP `applyTo(builder, values)` decorator pattern |
| D7 | Credential-rotation pool eviction **deferred** (documented limitation) | Test-phase scope; rotating a jdbc-password credential requires an engine restart to take effect. Tracked as a follow-up |

## Architecture

### 1. New engine credential type — `JdbcPasswordCredential`

`services/engine/.../action/credential/types/JdbcPasswordCredential.java`,
`implements DatabaseCredentialType`, Spring bean auto-registered by `CredentialRegistry`:

- `name = "jdbc-password"`, `displayName = "JDBC Username/Password"`
- fields: `username` (STRING, required), `password` (SECRET, required)
- `validate(values)`: both non-blank
- `applyCredentials(HikariConfig config, CredentialValues values)`: sets
  `config.setUsername(values.get("username"))` / `config.setPassword(values.get("password"))`.
  Does **not** construct or own the pool — the registry does.

Single generic type, not per-dialect: dialect/driver/URL are non-secret datasource config,
not credential data. (A future `PostgreSQLPasswordCredential` etc. could read `config.dialect()`
for `connectionInitSql`, but that is non-secret config and out of scope.)

### 2. Interface amendment (engine)

`DatabaseCredentialType` (shipped B.4, no impl yet) changes its single method:

```
// before (B.4)
DataSource buildDataSource(CredentialValues values, DatasourceRegistry.DatasourceConfigDto config);
// after (B.5)
void applyCredentials(HikariConfig config, CredentialValues values);
```

The `DatasourceRegistry.DatasourceConfigDto` parameter is dropped from the interface, breaking
the `credential -> infrastructure` import coupling (see §5).

### 3. Migration — `V24__datasource_credential_ref.sql` (admin)

Single migration, transaction-wrapped, **TRUNCATE before the NOT NULL add**:

```sql
BEGIN;
TRUNCATE tenant_datasource;
ALTER TABLE tenant_datasource DROP COLUMN encrypted_password;
ALTER TABLE tenant_datasource DROP COLUMN username;
ALTER TABLE tenant_datasource ADD COLUMN credential_ref VARCHAR(100) NOT NULL;
ALTER TABLE tenant_datasource ADD CONSTRAINT chk_credential_ref_pattern
    CHECK (credential_ref ~ '^[a-z][a-z0-9-]*$');
COMMIT;
```

- `credential_ref` is the **label** of a `jdbc-password` credential (distinct field, so one
  credential can back multiple datasources). No DB FK (composite credential key); referential
  integrity enforced in the application layer (see §4 delete guard).
- Fresh deploy: V1→V20(create)→V22(rename)→V23→V24; table empty, TRUNCATE is a no-op,
  `NOT NULL` add succeeds. No analogous V6 CASCADE risk (`tenant_datasource` has no inbound FKs).
- Live/dev incremental (V20–V23 applied): TRUNCATE empties the table first, so the `NOT NULL`
  add succeeds. **Requires an engine restart** (in-memory `DatasourceRegistry.configCache`
  holds the old DTO shape).
- Roadmap said "V24/V25" assuming automated backfill; a single V24 suffices under D2.

### 4. Admin service changes

- `TenantDatasource` entity: drop `username` + `encryptedPassword` (and their `@NotBlank`);
  add `credentialRef` with `@NotBlank @Pattern(^[a-z][a-z0-9-]*$)`.
- Remove `EncryptionService` field from `TenantDatasourceService` (class stays for
  `ConnectorService`/`TenantApiCredential`).
- `TenantDatasourceRequest`/`TenantDatasourceResponse`: drop `username`/`password`, add
  `credentialRef`.
- **DTO hygiene:** the engine-internal config is a **separate** record (no `password`,
  no `username`, carries `credentialRef` + non-secret config) — not the portal-facing
  `TenantDatasourceResponse` with a permanently-null password field.
- `testConnection(tenantId, ref)`: resolve credential values via
  `TenantCredentialService.resolvePath(tenantId, "jdbc-password", credentialRef)` →
  `VaultCredentialStore.read(vaultPath)` → `{username, password}`, then run the existing
  throwaway `DriverManagerDataSource` connection + SSRF/error-categorization logic unchanged.
  Tenant-scoped lookup makes this **IDOR-safe** by construction. Distinct error message when
  `credentialRef` does not resolve (stale ref) vs connection failure.
- `resolveForEngine`: return only non-secret config + `credentialRef` (no decrypted password).
- **Delete guard:** `TenantCredentialService.delete(tenantId, id)` queries
  `TenantDatasourceRepository` for rows with `credential_ref = entity.label AND
  tenant_id = tenantId`; returns **409 Conflict** naming the dependent datasource refs if any
  exist. Prevents dangling `credential_ref`.

### 5. Engine changes

- Extract the config record out of `DatasourceRegistry` into a standalone package-level
  record in `action.db` (e.g. `DatasourceConfig`): drops `username`/`password`, adds
  `credentialRef`. Both `DatasourceRegistry` and `DatabaseCredentialType` consumers depend on
  this shared type instead of nesting it in the `@Service`.
- `DatasourceRegistry.createPool(tenantCode, ref)`:
  1. fetch non-secret `DatasourceConfig` (incl. `credentialRef`) from admin (cached).
  2. `CredentialValues values = credentialRegistry.resolveForTenant("jdbc-password", tenantCode, config.credentialRef())`.
  3. `CredentialType t = credentialRegistry.get("jdbc-password")`; **guard**
     `if (!(t instanceof DatabaseCredentialType dbt)) throw new IllegalStateException(...)`.
  4. build `HikariConfig` with non-secret config + `poolName` (`werkflow-db-{tenant}-{ref}`) +
     `autoCommit`; `dbt.applyCredentials(hk, values)`; `new HikariDataSource(hk)`.
  - Registry keeps owning pool cache, circuit breaker, eviction. New `CredentialRegistry`
    dependency injected.
- Resolution failure surfaces as `CredentialResolutionException`, handled like HTTP delegates.
- No env-fallback for `jdbc-password` (always resolved with an explicit `credentialRef` via
  the three-arg overload). Explicitly documented to discourage adding one.
- Update stale Javadoc in `DatasourceRegistry` (class + config record) describing the old
  "admin decrypts, plaintext over channel" flow.

### 6. Documentation fixes (atomic with the PR)

`DatasourceController.resolveForEngine`, `DatasourceRegistry` class Javadoc, and
`TenantDatasourceService` class Javadoc currently describe AES decryption / `SecretsResolver` /
plaintext-over-channel. These are already partly aspirational (B.4) and become fully wrong the
moment V24 ships — update all three in the same PR.

## Error Handling

- Engine: missing credential / empty vault → `CredentialResolutionException` at pool creation;
  trips circuit breaker (existing behavior).
- Admin testConnection: stale/unresolvable `credentialRef` → distinct safe message; connection
  failure → existing categorized message (no topology/credential leakage).
- Delete guard: 409 with dependent datasource refs named.

## Known Limitations (documented, deferred)

- **Credential-rotation pool freshness (D7):** rotating a `jdbc-password` credential in OpenBao
  does not evict the engine's live Hikari pool; the engine continues using old credentials until
  restart. Compounded by the 60s `credentialPaths` metadata cache TTL. Auto-eviction
  (engine evict endpoint + admin-fired call on credential/datasource update) is a tracked
  follow-up (B.5-portal or B.6).
- **Datasource-update credential change:** changing a datasource's `credential_ref` likewise
  needs an engine restart to take effect (same deferral).

## Out of Scope

- Portal datasources-page credential picker → B.5-portal.
- Removing `EncryptionService` (still used by `ConnectorService`/`TenantApiCredential`).
- Automated AES→OpenBao backfill.
- Connector API-credential (`TenantApiCredential.secretValue`) migration → separate future phase.
- Credential-rotation pool eviction wiring (D7).
